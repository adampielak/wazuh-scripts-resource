CDB Lists

CDB stands for “constant database”. This acts as a high performance on-disk associative array, mapping keys to values. They allow only two operations: creation and reading. After the initial creation (compilation) of the CDB list, you can’t add any more data to that file.

Use cases:

  • Users (black or white lists)

  • IP address lookups

  • Indexing a list of bad domains

It’s also a convenient way to index databases and perform lookups on them.

CDB lists need to be stored in the directory /var/ossec/etc/lists/ and they need to have a specific format. For example, file /var/ossec/etc/lists/suspicious could contain entries like this:

key1:value
key2:value
key3:value

Each key has to be unique and must be followed by a colon and an optional value. The values between lines can be the same as long as the key is unique.

We need to define it in the ossec.conf file first by adding the following entry to the <rule> section of the ossec.conf file.

<ruleset>
        <list>etc/lists/suspicious</list>
</ruleset>

The last thing to do is to create new rules to trigger alerts whenever an event matches one of the entries in our new list.

<rule id="100001" level="10">
           <if_sid>***SOME RULE***</if_sid>
                <list field="srcip" lookup="address_match_key">lists/suspicious</list>
           <description>Suspicious IP found</description>
</rules>

This rule will always trigger if the source IP is found in our CDB “suspicious” list.

Full documentation for all types of Wazuh CDB list lookups:

https://documentation.wazuh.com/current/user-manual/ruleset/cdb-list.html |

Lab Exercise 5A - Distinguish between different ssh-watch-list values

For sshd invalid user login attempts where the attacker’s IP is in our custom ssh-watch-list file, cause a higher priority rule to fire than the standard rule 5710 and classifies attacker IPs as “bad” or “terrible”. Make sshd invalid user login attempts where the attacker’s IP is listed in our custom ssh-watch-list file with a value of “terrible”, cause an even higher priority rule to fire.

Example log entries

Jan 19 04:17:54 agent sshd[2365]: Failed password for invalid user blabla from 1.2.3.4 port 55969 ssh2
Jan 19 04:17:54 agent sshd[2365]: Failed password for invalid user root from 99.99.99.99 port 55969 ssh2
Jan 19 04:17:54 agent sshd[2365]: Failed password for invalid user admin from 10.10.10.10 port 55969 ssh2
Jan 19 04:17:54 agent sshd[2365]: Failed password for invalid user bill from 12.12.14.14 port 55969 ssh2

Initial Wazuh alert pattern for any of the above:

** Alert 1484799474.4188056: - syslog,sshd,invalid_login,authentication_failed,pci_dss_10.2.4,pci_dss_10.2.5,pci_dss_10.6.1,
2017 Jan 19 04:17:54 (linux-agent) 10.0.0.20->/var/log/secure
Rule: 5710 (level 5) -> 'Attempt to login using a non-existent user'
Src IP: 45.55.186.244
Jan 19 04:17:54 agent sshd[2365]: Failed password for invalid user blabla from 55.55.186.244 port 55969 ssh2

We want to make a special rule of higher severity fire if this rule 5710 is tripped by an attacker on our special list of IPs of concern (ssh-watch-list)

On you PC, create a file named ssh-watch-list which will be our CDB list (do not include a file extension such as .txt) and fill it with this content:

1.2.3.4:bad
55.55.186.244:bad
2.2.2.2:terrible
5.5.5.5:bad
10.10.10.10:terrible
23.32.23.32:bad

In your interface Navigate to Hamburger menu > Server management > CDB lists and click on the ‘Import files’ button. Drag and drop you CDB list file into the dialog box. You should see the ssh-watch-list show up in the list after refreshing.

Add to the <ruleset> section of the manager’s ossec.conf file:

<list>etc/lists/ssh-watch-list</list>

Add these rules to the local_rules.xml:

  <rule id="100112" level="10">
                <if_sid>5710</if_sid>
                <list field="srcip" lookup="address_match_key">etc/lists/ssh-watch-list</list>
                <description>SSH attacker on ssh-watch-list attempted to login using a non-existent user</description>
  </rule>

<rule id="100111" level="15">
   <if_sid>5710</if_sid>
   <list field="srcip" lookup="match_key_value" check_value="terrible">etc/lists/ssh-watch-list</list>
   <description>A truly terrible SSH attacker on ssh-watch-list attempted to login using a non-existent user</description>
</rule>

Feed the example log entries into wazuh-logtest and observe that those involving a listed IP fire rule 100112 instead of rule 5710, unless they have a value of “terrible”, in which case they fire the even more severe rule 100111.

Also, to see a great blog on importing the OSINT list of malicious IPs into a CDB:

https://wazuh.com/blog/cdb-lists

Active Response

In order to reduce risk and contain damage, we need in our system an automated remediation to security violations and threats. A remediation solution can perform actions or alerts configurations to prevent the access to the service or network where the threat came from.

This automated remediation is called “Active response” in Wazuh. Active response allows you to execute a script whenever a rule is matched by the ruleset. Any number of responses can be attached to any of the rules, but it is important to be careful.

A bad implementation of the rules and responses, can be dangerous. An attacker could use the rules against you.

By default the following active response scripts are already implemented with a default Wazuh installation:

  • disable-account: disables an account by setting “passwd -l” (Linux, Solaris, AIX only)

  • firewall-drop: adds an IP to the iptables deny list(Linux, Solaris, FreeBSD, AIX only)

  • firewalld-drop: adds an IP to firewalld drop list(Linux)

  • host-deny: adds an IP to the /etc/hosts.deny file (Linux, Solaris, FreeBSD, AIX only)

  • ip-customblock: Custom Wazuh block, easily modifiable for custom response

  • ipfw_mac: firewall-drop response script created for the Mac OS firewall

  • ipfw: firewall-drop response script created for ipfw, the FreeBSD IP packet filter

  • kaspersky`: Run an on-demand AV scan with Kaspersky (when installed on the monitored endpoint).

  • pf: firewall-drop response script created for pf, the BSD licensed stateful packet filter.

  • wazuh-slack: in order to post modifications

  • restart-wazuh: automatically restarts Wazuh when ossec.conf has been changed on UNIX systems

  • restart-wazuh.exe: automatically restarts Wazuh when ossec.conf has been changed on Windows

  • route-null: adds an IP to null route (only works on Linux and FreeBSD)

  • route-null.exe: adds an IP to null route

  • netsh.exe: uses Windows “netsh ipsec” to block an IP - for Windows Server 2012

When creating custom Active Response scripts one should be careful and should always test them before deploying them to production environments. These scripts are run as the root user, and thus perform any action an administrator can do.

Creating custom Active Commands:

The scripts need a way to be referenced by the ruleset to perform a particular active response. A command needs to be defined for this. Any executable placed in the agent’s active-response/bin/ folder can be used as an Active Response, the execd module will provide information of the alert through standard input to the executable, so in order to use information from the event triggering the executable must be able to read information from StdIn.

Finally, if the Active Response is intended to be stateful, that is, capable of running a secondary action after a given time, then when running it must write a stdout control message when executed providing the keys specific to the event. This is done so the agent will avoid executing the response multiple times while the secondary action is not been triggered.

Active response commands are defined in the ossec.conf file.

Command Values:

Tags

Values

Description

name

string

This provides an identifier for the active response command

executable

filename

Name of the executable to be run. This file must exist in the agents’ active-response/bin folder.

extra_args

string

Additional arguments to be read by active-response scripts.

timeout_allowed

yes/no

Specifies if the command has a secondary action to be taken after a given time. This is often used to undo the first action.

Example:

<command>
   <name>reboot</name>
   <executable>reboot.sh</executable>
   <timeout_allowed>no</timeout_allowed>
</command>

In this example, the command is called reboot and should execute the reboot.sh script located at /var/ossec/active-response/bin. It is not able to timeout and the script doesn’t expect any input. The script can be as simple as:

#!/bin/sh
reboot

This would reboot the endpoint when triggered.

Let’s see a stateful example from the stock active responses:

<command>
   <name>host-deny</name>
   <executable>host-deny</executable>
   <timeout_allowed>yes</timeout_allowed>
</command>

In this case the command is called “host-deny”, and should execute the host-deny, the script needs the srcip and the command is able to timeout.

While active responses can be written in any language, handling JSON can be tricky in Bash and interpreted languages like Python may not be able to run in all endpoints as they would depend on a local installation of the interpreter.

Creating custom Active Response:

Now we need to bind the command to one or more rules or a specific severity level, which creates an active response. In the active response definition, it is declared when a command is going to be executed and where, meaning on which environment (Agent, Manager, Local, or everywhere). The following tags can be used for active responses:

Tags

Value

Description

disable

yes/no

If set to yes it will disable active responses

command

Must reference a previously declared command

Name of the command to be used

location

local, server, defined-agent, all

Endpoints on which the active response will be executed. local is the machine from which the event originated, server is the Wazuh manager, all is all agents (this excludes the manager).

agent_id

Numerical ID of agent

When defined-agent is used in location,

level

0-15

Rule level threshold as a criteria to trigger active response

rules_id

comma-separated rule IDs

Rules that will trigger the active response when matched

rules_group

comma-separated rule group names

Rule groups that will trigger the active response

timeout

Number

Amount of time (in seconds) after initial active response is triggered before the secondary action is taken

repeated_offenders

comma-separated numbers

Amount of time (in minutes) that active response is increased for repeating offenders

Example:

<active-response>
   <command>reboot</command>
   <location>server</location>
   <rules_id>105002</rules_id>
</active-response>

In this case, the reboot command will be executed on the Wazuh manager everytime custom rule 105002 is fired.

Another example:

<active-response>
   <command>host-deny</command>
   <location>local</location>
   <level>7</level>
   <timeout>600</timeout>
</active-response>

In this example the host-deny command will be executed on the agent that reported the rule-matching event, with rule levels equals or bigger than 7. This response has 600 seconds for the timeout.

Stateful vs stateless command:

It’s important to understand the difference between a command being stateful and a command being stateless. In our examples we saw one of each.

A Stateful command is a command that has the capability to remove the applied action after a certain period of time.

To configure a command like this, we need to set the timeout_allowed option in the command to yes, and then configure a timeout value in the active response to indicate how long to wait until the applied action is automatically reversed.

A Stateless command is that command that doesn’t have the capability of removing the applied changes that the response applied.

An example for a Stateful command is the host-deny command from above.

Lab Exercise 5B - Deploy and test Linux firewall active response

Configure Wazuh to make you Linux agents firewall-block ssh brute force offenders.

  1. On the manager, edit the ossec.conf file, find the following comment:

<!-- Files to monitor (localfiles) -->

And above it add the following:

<active-response>
   <command>firewall-drop</command>
   <location>local</location>
   <rules_id>5712</rules_id>
   <timeout>120</timeout>
</active-response>

The command “firewall-drop” referenced above is already defined in ossec.conf on the manager here as seen here below. You do not need to add it in.

<command>
      <name>firewall-drop</name>
      <executable>firewall-drop</executable>
      <timeout_allowed>yes</timeout_allowed>
</command>

When invoked by an active response, this command will call the “/var/ossec/active-response/bin/firewall-drop” script on the agent, passing an IP number to it to be blocked via the local iptables firewall. Then after a timeout period it will be called again to un-firewall the offending IP. If you wish, take a moment to examine the source of this script. It can serve as an excellent template for you to build your own custom active responses later.

  1. Restart the Wazuh manager to apply this change.

  2. From your linux-agent, verify that you can ping the indexer agent:

ping indexerA6.lab.wazuh.info

  1. perform an ssh dictionary attack against your indexer agent using hydra that was installed during exercise 2A as well as the password list created then:

hydra -l badguy -P pass.list indexerA6.lab.wazuh.info ssh -I

  1. verify that you have now lost connectivity to the indexer agent from the linux agent by performing another ping to it:

ping indexerA6.lab.wazuh.info

  1. After the 120 second timeout passes, we should also be able to see the record of the active response being performed and then later reversed on your linux-agent:

    • Inspect the AR log with Wazuh dashboard with this search:

      decoder.name:"ar_log_json"

  2. Confirm that your linux-agent is again able to reach your indexer system.

ping indexerA6.lab.wazuh.info

  1. Check Wazuh dashboard later today for active response activity (see step 6) and you will probably see real attackers in the wild being blocked by one or more of your Linux lab systems.

Alternatively you may also configure the active response on Windows:

  1. On the manager add the following configuration after the <command> section:

<active-response>
  <command>netsh</command>
  <rules_group>authentication_failures</rules_group>
  <location>all</location>
  <timeout>120</timeout>
</active-response>

Note that this will trigger the netsh command on all endpoints whenever an event of the authentication_failures group is observed. This will only be executed on endpoints that have this command (Windows) but it will be done even if the event is observed on a different endpoint.

  1. Restart the Wazuh manager to apply this change by clicking on the restart button of the web interface.

  2. From the linux-agent, verify that you can connect to the Windows 3389 port:

nc -tvz windows-agentA6.lab.wazuh.info 3389

If successful you should see:

 Connection to windows-agentA6.lab.wazuh.info 3389 port [tcp/*] succeeded!

  1. Given that the hydra RDP module has been reported to be unreliable we can verify the active response by attacking the indexer agent instead:

hydra -l badguy -P pass.list indexerA6.lab.wazuh.info ssh -I

  1. Verify that you have lost connectivity to the windows agent:

nc -tvz windows-agentA6.lab.wazuh.info 3389

After some time you should see the command time out:

  nc: connect to windows-agentA6.lab.wazuh.info port 3389 (tcp) failed: Connection timed out

Alternatively you may block your own IP by failing to log via RDP into Windows multiple times.
