What is Wazuh?

  • A situational awareness tool to monitor your electronic assets.
  • An important resource for achieving Security Regulatory Compliance.

Main Components

  • Log Collection.
  • Log Analysis (customizable set of over 4300 alert rules).
  • Alert/Event Storage (Wazuh Indexer document database).
  • Web Dashboard (Wazuh Dashboard).
  • File Integrity Monitoring.
  • System Inventory and Vulnerability Detection.
  • Security Configuration Assessment.
  • Integrations, Active Response, and Email.

Main Components

19

Network Communications












  • Agent-Manager connections are compressed and encrypted with per-agent pre-shared keys (AES) over TCP or UPD 1514.
  • The Remoted daemon can directly accept TCP and/or UDP port 514 messages from syslog-sending devices.
  • For a more robust Syslog collection, Syslog server(s) can be used on agent(s) or the manager.
20

Wazuh Secure Communication

  • All communication between Wazuh components is secured and encrypted using AES and TLS.
  • The Wazuh Server Worker nodes use TLS to sync configuration and state data with the Master node.
  • Each Wazuh agent is assigned a unique key for reporting to the manager.
  • Significant privilege separation and isolation is built into the solution.

Having said this, access to the Wazuh central components should be limited since:

  • They centralize priviledged security information of all monitored assets.
  • The Wazuh manager can trigger Active Responses or other remote commands if enabled.

Agent Enrollment

25

Authd Daemon

  • The Wazuh Agents must complete the enrollment process with the Wazuh Server before they can start reporting in.
  • These enrollment requests are handled by the authd daemon running on the Wazuh Servers.
  • In a clustered environment, the enrollment requests are processed by the Master node ONLY. Worker nodes can receive them but they will be forwarded to the Master node.
  • Using password authentication for the enrollment process is not required but it is recommended.
  • It is also possible to use certificates to validate agent/manager identity.
26

Agent Enrollment Methods

  1. With agent installer via deployment variables.
  2. Using agent-auth tool.
  3. Agent auto-enrollment.
  4. Requesting registration directly from Wazuh API (rare).

Remotely Upgrading Wazuh Agents

29

Ways To Remotely Upgrade Wazuh Agents

Wazuh agent upgrades can be pushed from Wazuh Servers to outdated agents that are currently active. The same connection used for communication between the agent and the Wazuh Server will be used to send the upgrade package.

  • Wazuh API via Wazuh Dashboard:
    • From the Endpoints Summary Dashboard in the Server management section, you can select agents and upgrade them.
    • By going to Server management -> Dev Tools users may interact with the API to initiate the agent upgrade process.
  • Wazuh API directly:
    • Automatically routes upgrade tasks to the right Wazuh Servers. This could be used to Invoke regular push upgrades of connected outdated agents via a custom script/cron job.

General Configuration

32

The “ossec.conf” File

  • It is the main configuration file for Wazuh Servers and agents.

  • File location:
    • Linux (Agent/Manager): /var/ossec/etc/ossec.conf
    • Windows (Agent): C:\Program Files (x86)\ossec-agent\ossec.conf
    • MacOS (Agent): /Library/Ossec/etc/ossec.conf

  • This file contains the configuration for:

    • Log collection and analysis.
    • File integrity monitoring (Syscheck).
    • Active response.
    • Wazuh integrations.
    • Cluster, Agent enrollment/connection.
    • Agent buffer (Leaky bucket).
    • Other capabilites such as: email notifications, alerts threshold, etc.
33

The “internal_options.conf” File

Low-level configuration file for agents. It is mainly used for debugging and troubleshooting issues.

  • File location:
    • /var/ossec/etc/internal_options.conf

    It contains all the available options, but it is overwritten during Wazuh upgrades.

    • /var/ossec/etc/local_internal_options.conf

    Used to keep customized options during upgrades.

  • These internal options allow:

    • Setting the verbosity level for specific modules’ logs.
    • Enabling remote commands.
    • Other settings for specific purposes.
34

The “agent.conf” Group Configuration File

By creating different agent groups and adding agents to these groups, we can manage their configuration centrally from the Wazuh Server or from the Wazuh Dashboard. These configurations are pushed to the agents and they take precedence over the agents’ own configuration file (“ossec.conf”).

  • File location: /var/ossec/etc/shared/<GROUP_NAME>/agent.conf.
  • Agents are automatically restarted after receiving the group configuration.
  • Agents can belong to multiple groups.
  • If no group is specified when enrolling the agent, they will be added to the “default” group.
  • Group names are case sensitive and they can’t include spaces.
  • Specific agent config sections are possible on a per-OS, per-profile, and per-agent basis, allowing great flexibility.
35

Agent Groups and Profiles

When not all agents from the same group should apply the same configurations, you can specify different agent_config blocks.

The agent_config configuration block accepts the following options:

  • os only agents on specific operating systems will use this block.
  • profile only agents with specific profiles defined in client configuration will use this block.
  • name only agents with specific names will use this block.
36

Agent Profile Example

In the agent’s ossec.conf, the <config-profile> setting can include multiple profiles:

Local ossec.conf file on the Wazuh Agent.

Log Analysis and Regulatory Compliance

  • Computer-aided log analysis is a potent tool for identifying threats or potential problems within a vast stream of collected log events.
  • Many regulatory compliance requirements mandate regular review of security logs.
  • This task is often impractical and unsustainable without the assistance of machine analysis and categorization.
  • It aids in classifying which events should be retained to meet regulatory requirements, leading to reduced storage costs.

Log Analysis with Wazuh

Wazuh’s log analysis engine is capable of:

  • Using decoders to extract key values from a log message into named fields.
  • Analyzing the decoded log message against a heirarchy of alert rules.
  • Generating alerts according to the matching rule level.
  • Triggering other real time actions in response to specific alerts.
45

Log Flow












  • The wazuh-logcollector module on the Wazuh Agent collects the logs.
  • The wazuh-analysisd module on the Wazuh Server analyzes the log entries.
  • The wazuh-execd module is used for executing Active Responses.
  • The wazuh-maild module sends alerts via email.
46

Stages of Log Analysis

  • Log collection on agents is defined using <localfile> sections
    • The <log_format> field sets the format of the log to be read.
    • The <location> field specifies the source of the log.
  • Pre-decoding
    • Extracts fields based on the <log_format> value, like program_name from the Syslog header.
  • Decoding
    • Extracts program-specific fields like srcip or username.
  • Rule-based analysis of the decoded log
    • One or more instances and types of matching criteria can be compared against individual log fields or the whole log.
    • Matching field values to CDB lists also supported.
47

Examples of <localfile> Sections

Collecting a Windows EventChannel, a Syslog file, and a wildcarded set of JSON files:

<localfile>
  <log_format>eventchannel</log_format>
  <location>Application</location>
</localfile>

<localfile>
   <log_format>syslog</log_format>
   <location>/var/log/messages</location>
</localfile>

<localfile>
  <log_format>json</log_format>
  <location>/var/log/suricata/eve-*.json</location>
  <label key="@source">suricata</label>
</localfile>
48

<localfile> Sections Using Commands

Configuration to collect the output of a command:

<localfile>
   <log_format>full_command</log_format>
   <command>netstat -natp | grep '^tcp .*LISTEN ' | sort | sed 's/LISTEN \+[0-9]\+\///g'
   | grep -v "127.0.0.1:" | awk '{print $4"\t"$6}'</command>
   <frequency>300</frequency>
   <alias>netstat listening ports</alias>
</localfile>

Related rule:

<rule id="533" level="7">
  <if_sid>530</if_sid>
  <match>ossec: output: 'netstat listening ports</match>
  <check_diff />
  <description>Listened ports status (netstat) changed (new port opened or closed).
  </description>
  <group>pci_dss_10.2.7,pci_dss_10.6.1,gpg13_10.1,gdpr_IV_35.7.d,hipaa_164.312.b,
  nist_800_53_AU.14,nist_800_53_AU.6,tsc_CC6.8,tsc_CC7.2,tsc_CC7.3,</group>
</rule>

Wazuh Indexer, Dashboard and Filebeat
The Wazuh Indexer is a highly scalable full-text search and analytics engine, that receives data shipped by Filebeat, indexes it and allows the Wazuh dashboard
access as its primary data source. It is a version of OpenSearch built and maintained by the Wazuh team.
The Wazuh Dashboard is the web front-end to the data in the Wazuh indexer. This customized package forked from OpenSearch dashboards includes the Wazuh plugin which on top of solution specific dashboards interacts with the Wazuh API to provide visibility and control over the solution’s configuration and status.
Filebeat belongs to the Beats family of data shippers. It is used to ship Wazuh’s alerts.json or archives.json records to the Wazuh Indexer. It also can do some data transformation and/or enrichment, but generally those are best done at the ingest pipeline level on the Wazuh Indexer side.
