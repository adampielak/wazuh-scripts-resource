Wazuh Decoders
==============


.. admonition:: Definition

        Decoders are designed to extract any relevant data from events.

* Decoding an event is done in two phases:
        - **Predecoding:** extracts predefined fields from known log headers, such as the program name from a syslog entry. This can be used to select an appropriate family of decoders.

        - **Decoding:** extracts specific field values from the event, for example: IP address or usernames.

Wazuh rules commonly use matching criteria that refers to one or more specific fields extracted by decoders, rather that matching for general patterns across the whole event. This improves accuracy and performance.

Regular Expressions in Wazuh
============================

* The <prematch> and <regex> options present in stock Wazuh decoders use a simple high-performance regular expression (regex) syntax:

.. code-block:: none
  :caption: OS RegEx

  \w -> A-Z, a-z, 0-9, '-', '@', '_' characters
  \d -> 0-9 characters
  \s -> Spaces " "
  \t -> Tabs
  \p -> ()*+,-.:;<=>?[]!"'#$%&|{}
  \W -> Anything not w
  \D -> Anything not d
  \S -> Anything not s
  \. -> Anything


* Starting with Wazuh 4.1.0, you can optionally use full PCRE2 expressions by specifying <prematch type="pcre2"> and <regex type="pcre2">.  For details on using the old and new Wazuh regex expression formats, see: `Documentation <https://documentation.wazuh.com/current/user-manual/ruleset/ruleset-xml-syntax/regex.html>`_


Simple decoder example

Let's write a simple decoder for this log event:


Example of a simple log event:

        .. code::

                2016-04-29T10:01:04.600374-04:00 manager myprogram[9123]: test connection
                from 192.168.1.1 via test-protocol1


Example of our new decoder:

        .. code::

                <decoder name="myprogram">
                  <program_name>myprogram</program_name>
                </decoder>


Child Decoder
=============

Example of our Child decoder:

        .. code::

                <decoder name="myprogram-test-connection">
                  <parent>myprogram</parent>
                  <prematch>^test connection </prematch>
                  <regex offset="after_prematch">^from (\S+) via (\S+)$</regex>                  <order>srcip, protocol</order>
                </decoder>


We're referencing the decoder from before "myprogram" and adding new parameters to extract the SRC IP address and the protocol field.

Wazuh Dynamic Decoders
======================


Windows Eventchannel Dynamic Decoder
====================================

* The Wazuh Windows' Agent collects Eventchannel events using the full Windows Event Schema.
* The *windows-eventchannel* decoder will automatically extract all available fields from these events.
* Default Windows rules include the **no_full_log** option since all available fields are already parsed.
* The Ruleset Test or wazuh-logtest capabilities cannot be used to test these events (a few workarounds can be used for this).

JSON Dynamic Decoder
====================

* In addition to analyzing plain text logs and Windows event channel logs, Wazuh Server can also take JSON-formatted logs as input:
        - Just like the *eventchannel* decoder, the *JSON* decoder is also able to automatically extract all available fields from the event.

        - Wazuh can process JSON log records, utilizing the predefined fields within the records for rule evaluation and displaying them in the Wazuh dashboard.

        - Many applications, such as Snort and Suricata IDS engines, and most cloud service providers can write alerts to JSON files.

Wazuh Rules

Wazuh Rules
===========

* Rules files locations:
        - **/var/ossec/ruleset/rules/** - for Wazuh's default rules.

        - **/var/ossec/etc/rules/** - for custom rules.

* Rules use the fields extracted by the decoders from the events to generate alerts when their conditions are met.

* Custom rules ID numbers should be kept in the range from 100,000 to 120,000 to avoid conflicts with the official Wazuh Ruleset.

* Rule severity levels range between 0 to 15 (0 being the lowest).

Wazuh Rules (Continued)
=======================

* There are two types of rules:
        - **Atomic Rules** will match based on a single event, without any correlation.
                - Examples: Authentication failure, alert on unique event, etc.
        - **Composite Rules** are triggered by a sequence of events. Easily recognizable by the parameters:
                - **frequency**: specifies how many times an event or pattern must occur.

                - **timeframe**: determines how far back (in seconds) it should search for the matching events.

                Composite rules look for **frequency** events in the last **timeframe** seconds.


Rule Severity Levels
====================

.. table::
  :class: long-table

  +-------+----------------------------------+---------------------------------------------------------------------------+
  | Level | Title                            | Examples                                                                  |
  +=======+==================================+===========================================================================+
  | 0     | Ignored                          | No action taken. Can be used to avoid false positives.                    |
  +-------+----------------------------------+---------------------------------------------------------------------------+
  | 2     | Low priority notification        | System notification with no security relevance.                           |
  +-------+----------------------------------+---------------------------------------------------------------------------+
  | 3     | Authorized events                | Successful login attempts, firewall allow events, etc.                    |
  +-------+----------------------------------+---------------------------------------------------------------------------+
  | 4     | System low priority error        | Errors related to bad configurations or unused applications.              |
  +-------+----------------------------------+---------------------------------------------------------------------------+
  | 5     | User generated error             | Missed passwords, denied actions, etc.                                    |
  +-------+----------------------------------+---------------------------------------------------------------------------+
  | 6     | Low relevance attack             | An attack that does not affect the system.                                |
  +-------+----------------------------------+---------------------------------------------------------------------------+
  | 7     | "Bad word" matching              | Events that have words like "bad", "error", etc.                          |
  +-------+----------------------------------+---------------------------------------------------------------------------+
  | 8     | First time seen                  | First time an IDS event is fired or the first time an user logged in.     |
  +-------+----------------------------------+---------------------------------------------------------------------------+
  | 9     | Error from invalid source        | Attempts to login as an unknown user or from an invalid source.           |
  +-------+----------------------------------+---------------------------------------------------------------------------+
  | 10    | Multiple user errors             | Multiple bad passwords, multiple failed logins, etc.                      |
  +-------+----------------------------------+---------------------------------------------------------------------------+
  | 11    | Integrity checking warning       | Modification of binaries or the presence of rootkits.                     |
  +-------+----------------------------------+---------------------------------------------------------------------------+
  | 12    | High relevance event             | Error or warning messages from the system, kernel, etc.                   |
  +-------+----------------------------------+---------------------------------------------------------------------------+
  | 13    | Unusual error                    | Most of the times it matches a common attack pattern.                     |
  +-------+----------------------------------+---------------------------------------------------------------------------+
  | 14    | High security event              | Most of the times done with correlation and it indicates an attack.       |
  +-------+----------------------------------+---------------------------------------------------------------------------+
  | 15    | Severe attack                    | Immediate attention is necessary.                                         |
  +-------+----------------------------------+---------------------------------------------------------------------------+

Composite rules design scope
============================

* The Wazuh analysis engine is optimized for high performance.

* Aproximately 7% of all stock Wazuh rules are composite.

* Composite rules allow the Server to recognize repeated activity over short durations.

* The correlation is only done per Server and for the most recent 8192 events.

* For broader correlation you may use: 

  *  The indexer's Alerting capability.

The Wazuh Ruleset
=================

Ruleset Components
==================
 
There are three categories of items that make up the full ruleset that the Wazuh Server uses to analyse log events.

* Decoders:

        - Stock decoder files, located at */var/ossec/ruleset/decoders/*.

        - Local (custom) decoder files, located at */var/ossec/etc/decoders/*.

        - Decoders are grouped in files by family with parent-child relationships, and are identified by decoder name.


Ruleset Components
==================

* Rules:
 
        - Stock rule files can be found in */var/ossec/ruleset/rules/*.

        - Local (custom) rule files can be found in */var/ossec/etc/rules/*.

        - Rules are identified by a unique rule id number, and grouped in files by family with parent-child relationships.

* Lists:
 
        - CDB (Constant Data Base) lists are referenced by various rules as matching criteria.

        - Stock and local lists are stored in individual files in */var/ossec/etc/lists/*.


The Ruleset in the Server's *ossec.conf*
=========================================

This section of **ossec.conf** on the Wazuh Server identifies the various sources making up your full ruleset, with the locations always relative to the **/var/ossec/** directory.


.. code:: 

  <ruleset>
    <!-- Default ruleset -->
    <decoder_dir>ruleset/decoders</decoder_dir>
    <rule_dir>ruleset/rules</rule_dir>
    <rule_exclude>0215-policy_rules.xml</rule_exclude>
    <list>etc/lists/audit-keys</list>
    <list>etc/lists/amazon/aws-eventnames</list>
    <list>etc/lists/security-eventchannel</list>
    <!-- User-defined ruleset -->
    <decoder_dir>etc/decoders</decoder_dir>
    <rule_dir>etc/rules</rule_dir>
  </ruleset>

Updating the Stock Wazuh Ruleset
================================

* The Wazuh default ruleset is updated during a Server upgrade.

* Future ruleset content may be downloaded and added as custom rules and decoders.

* Important notes:

  * Some syntax of future versions may not be understood by an older Server.
   
  * After upgrading, you must remove the custom rules to avoid duplicate content.

Wazuh encourages everyone to contribute new rules & decoders to the Wazuh GitHub repo.

Ruleset Propagation Across the Server Cluster
=============================================

* **Rules and Decoders**
        - The *stock* decoders and rules are not propagated from the master node to the worker nodes.

        - Local (custom) decoders and rules files are propagated from the master node to the worker nodes.

* **Lists**
        - All list files (stock or local) in */var/ossec/etc/lists/* are propagated from the master node to the worker nodes.

* **Server restart requirement**
        - Though local rules and decoders, and all lists are rapidly propagated across the cluster, servers do not automatically restart to reload the newly changed ruleset files.

File Integrity Monitoring
=========================


File Integrity Monitoring (FIM)
===============================

How does FIM work? 

FIM periodically calculates and records the unique cryptographic checksum and other attributes of specified files, and alerts when they are modified.

* In Wazuh, the Syscheck module performs FIM.
* FIM is a key requirement for PCI DSS (Req. 11.5).
* In addition to checksums, Syscheck also watches for changes to file ownership, size, POSIX file permissions, and Windows ACLs.


File Integrity Monitoring (FIM)
===============================

1. The Wazuh agent scans security relevant areas of the file system.
2. Checksums and other attibutes are stored locally if they didn't previously exist.
3. Attributes of previously seen objects are compared.
4. If there is any difference a FIM event is sent to the Wazuh Server.

* Locally stored information is synchronized with the Server for inventory purposes.
* A copy of monitored text files can be mantained to provide *diff* (change details) output in the events.

  * This is limited by default to 50MB per file and 1GB in total.


Wazuh Syscheck options for FIM
==============================

Wazuh Syscheck options 

.. code::

        <syscheck>
                <frequency>21600</frequency>
                <directories realtime="yes" check_all="yes">/usr/bin,/usr/sbin</directories>
                <directories whodata="yes" check_all="yes">/etc,/bin,/sbin</directories>
         </syscheck>


* **frequency** sets the frequency of a Syscheck run, measured in seconds.
* **realtime** enables the realtime or continuous monitoring on Linux and Windows systems.
* **check_all** checks the md5/sha1/sha256 hashes, size, owner, group, and permission of files.
* **report_changes** includes a diff report of changes to text files.
* **whodata** reports on the user and process that actually performed the file change. This uses its own special realtime engine so when whodata is enabled, there is no need to use the *realtime* option.

Notes on FIM Scheduled Scans After an Agent Restarts
====================================================

* FIM scans are performed every 12 hours by default.

* FIM database is discarded when the agent is restarted.

* Changes that ocurred after a scan and before a restart are not reported.

* This is a design choice to reduce the amount of alerts during system upgrades.

* Using **realtime** or **whodata** will provide guaranteed visibility.

Rootkit Detection
=================


What is a Rootkit?
==================

.. admonition:: Definition

  A rootkit is a program that can hide itself as well as running processes, file or network connections from the host where it's running. 

  It can change access rights of files and directories.

* aim:  to run "incognito" in the background for as long as possible so that an intruder has access to the system.

Kinds of Rootkits?
==================

There are two kinds of rootkits:

        - **User-mode (user level) rootkit:** covertly replaces common UNIX binaries or libraries with infected versions to hide its existence and to gain root privileges if needed.

        - **Kernel-mode (system level) rootkit:** modifies or replaces the kernel.

===================================
How to detect a Rootkit with Wazuh?
===================================


Detecting Rootkits
==================

* **Syscheck Module:** The Wazuh FIM monitors changes of files such as binaries that a user level rootkit might replace.
* **Rootcheck Module:** Wazuh scans for inconsistent reporting across system calls about processes, sockets, files, etc...

==================================
